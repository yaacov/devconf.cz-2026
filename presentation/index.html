Language Models

Language Models

Base Model, Text Completion

from transformers import AutoModelForCausalLM, AutoTokenizer

model_path = "ibm-granite/granite-4.1-8b-base"
tokenizer = AutoTokenizer.from_pretrained(model_path)
model = AutoModelForCausalLM.from_pretrained(model_path, device_map="auto")

prompt = "If you want to migrate a virtual machine to KubeVirt,"
outputs = model.generate(tokenizer(prompt, return_tensors="pt").to("mps"))
print(tokenizer.decode(outputs[0]))
# Output: " you must first prepare the source environment.
#   This book covers migrations from VMware vSphere ..."
Goal: Show that base models are raw sentence completion engines running locally in system memory. Let's start by querying a raw base model using a simple Python script. No API wrapper, no servers — just direct weight execution. We load IBM's Granite 4.1 8B Base model into local memory using transformers and send an unfinished sentence as a prompt. The weights don't recognize an instruction; instead, they compute the mathematically most likely sequence continuation based on internet text. The response reads like a paragraph from a documentation guide or book, which is exactly the pattern of raw data the base model was trained on.

Instruct Model

Fine-Tuned to Follow Instructions

chat = [{"role": "user", "content": "How do I list VM migrations?"}]
prompt = tokenizer.apply_chat_template(chat, add_generation_prompt=True)
print(prompt)
# <|start_header|>user<|end_header|>
# How do I list VM migrations?<|eot|>
# <|start_header|>assistant<|end_header|>
outputs = model.generate(tokenizer(prompt, return_tensors="pt").to("mps"))
print(tokenizer.decode(outputs[0]))
Goal: Demonstrate how instruction tuning redirects sequence prediction into problem-solving. Now let's run the instruct-tuned variant of the exact same Granite 4.1 model. In our Python code, we introduce structure. We wrap our query in a list of role-based messages and run it through the tokenizer's chat template. This chat template injects system control tokens (like start_header_id) to construct syntactic boundaries. The model detects the instruction structure and, instead of continuing the sentence, outputs a direct terminal command to answer our question. Now we see how models handle instructions. But running raw Python files is heavy and stateful. How do we wrap this in a network API? Let's move to LM Studio.

[ Model Weights ]
       ▼

Goal: Explain the architectural gap bridged by local/remote inference server wraps. While running raw Python scripts is great for testing weights, it's highly stateful, heavy, and complex to integrate into applications. We need to decouple our application layer from our local machine memory lanes. This is what an Inference Server (like LM Studio or vLLM) does. It wraps the raw model-loading mechanics in a stateless, highly optimized HTTP REST server that speaks the standard OpenAI JSON API. Let's test this transition live using curl against our local inference server to demonstrate how stateless JSON payloads drive our tools.
Inference Servers

Conversational Mode

What Happens Without Tools?

curl http://localhost:1234/v1/chat/completions -d '{
  "messages": [
    {"role": "user", "content": "Fetch migration plans inside the cluster"}
  ],
  "tools": []
}'
# Output: "I cannot access the cluster directly.
#   Please run: kubectl mtv get plans"

* The /v1/chat/completions endpoint is the OpenAI-compatible API — the de facto standard spoken by LM Studio, vLLM, Ollama, and others.

Goal: Show how the aligned model reacts with text instructions when no external tool system is advertised. Notice the URL: /v1/chat/completions. This is the OpenAI-compatible API — the de facto standard that virtually every inference server (LM Studio, vLLM, Ollama, etc.) now speaks. We're hitting a local model, but the JSON shape is identical to what you'd send to OpenAI. Here, we query our model through the inference wrapper. Note that we explicitly pass an empty "tools": [] array in the JSON body. The model processes our conversational intent ("Fetch migration plans"), but it knows its limits: it is text-bound and cannot interact with systems directly. It answers in plain Markdown, instructing us (the human operator) to perform the terminal command.

Tool Usage

The Request — Including a List of Available Tools

curl http://localhost:1234/v1/chat/completions -d '{
  "messages": [{"role": "user", "content": "Fetch migration plans"}],
  "tools": [{"type": "function", "function": {
    "name": "mtv_read",
    "description": "Query MTV resources (read-only)",
    "parameters": {
      "properties": {"command": {"type": "string"}, "flags": {"type": "object"}},
      "required": ["command"]
    }
  }}]
}'
Goal: Show how exposing a tools schema triggers a structural shift in the model's generated response. Now we send the exact same prompt, but we populate the "tools" array with a schema describing our mtv_read function. Because the model is trained to recognize schemas, its attention layers shift. Instead of returning conversational text directing the user, the model will output a machine-readable JSON execution request. Let's inspect the payload returned by the server.
Inference Servers

Tool Usage

The Response — A Structured Tool Call Instead of Text

{"choices": [{
  "message": {
    "role": "assistant",
    "content": null,
    "tool_calls": [{
      "function": {"name": "mtv_read",
                   "arguments": "{\"command\": \"get plan\"}"}
    }]
  },
  "finish_reason": "tool_calls"
}]}
Goal: Deeply analyze the JSON tool execution payload returned by the model. Look at this response payload. This is the magic of structured alignment. Note that "content" is entirely null. The model did not output any conversational response. Instead, it returned a structured "tool_calls" block, populated "arguments" as valid JSON containing "resource": "plans", and set "finish_reason" to "tool_calls". The model is no longer talking to us — it is calling our code!

Your First MCP Server

Implementing the Tool Call

from fastmcp import FastMCP
import subprocess

mcp = FastMCP("mtv")

@mcp.tool()
def get_plan(namespace: str = "default") -> str:
    """Get migration plans from the cluster"""
    return subprocess.run(
        ["kubectl", "mtv", "get", "plan", "-n", namespace],
        capture_output=True, text=True).stdout

mcp.run(transport="sse", port=8081)

* MCP (Model Context Protocol) standardizes how tools are discovered and called, regardless of model or server.

Goal: Show how simple it is to implement the tool the model just called. The model generated a tool_calls payload for mtv_read. Now we show how to build the server that handles it. With FastMCP, you decorate a Python function with @mcp.tool() and the framework generates the JSON schema, handles SSE transport, and advertises the tool to any connecting agent.

Bash Tool vs. MCP Server

One Shell to Rule Them All, or Dedicated Servers?

# Bash tool: the agent runs arbitrary commands
bash -c "kubectl mtv get plan | grep -i fail"
# → unrestricted, unpredictable, no schema
// MCP server: the agent calls a typed function
{"method": "tools/call",
 "params": {"name": "mtv_read",
            "arguments": {"command": "get plan"}}}
// → sandboxed, discoverable, self-describing
Goal: Contrast raw shell execution against the structural safety and discovery patterns of MCP using software engineering principles. Execution Interface: Exposing a raw Bash tool to an LLM is like calling eval() on raw, untrusted user strings. An MCP server acts like an RPC interface with a strict, type-safe API contract. Memory Safety: Standard shell execution is like raw pointer arithmetic. An MCP tool call is like object-oriented encapsulation; the internals are private. Security Sandbox: Running a generic shell gives the LLM "Ring 0" system access. An MCP server runs on capability-based security under least-privilege rules. Service Discovery: To learn shell utilities, an agent must scrape help manuals. An MCP server exposes self-describing endpoints (tools/list) for instant discovery.

Adding MCP Support to My CLI

[ kubectl-mtv ]
      │
  ┌───┴───┐
  ▼       ▼
[CLI]   [MCP Server]
Goal: Highlight the architectural shift of embedding MCP runtimes directly inside existing Go-based CLI tooling. A major development in CLI design is embedding the MCP server directly inside the original binary. kubectl-mtv is built this way. The exact same executable serves two roles. Run standard commands like kubectl mtv get plans and it outputs human-readable tables. Run kubectl mtv mcp-server --sse --port 8081 and it enters background daemon mode, spinning up an HTTP server that speaks Model Context Protocol over Server-Sent Events (SSE). No external node scripts, no separate docker images, and no integration drift. Let's trace how the HTTP connection behaves in real-time.

So You Want to Implement MCP?

Add Custom Code to Your CLI, or Just Automate Using --help

# Scan the CLI's --help output → generate MCP tools
cli2mcp scan kubectl-mtv -o mtv.tools.json

# Start an MCP server exposing those tools
cli2mcp serve mtv.tools.json
{"command": "kubectl-mtv", "tools": [
  {"name": "get",    "description": "Get resources",
   "args": [{"name": "resource", "type": "string"}]},
  {"name": "create", "description": "Create resources", ...},
  {"name": "health", "description": "Check MTV health", ...}
]}
Goal: Show the fastest path from an existing CLI to MCP — two commands. cli2mcp is a Python library that scans a CLI tool's --help output, extracts its arguments and subcommands, and generates a JSON descriptor. It then serves that file as a fully functional MCP server. Step 1: cli2mcp scan reads kubectl-mtv --help, walks every subcommand, and writes a tools JSON file with names, descriptions, and argument schemas. Step 2: cli2mcp serve starts a stdio-based MCP server. Every entry in the JSON becomes a callable tool. When the model calls "get", cli2mcp runs subprocess.run(["kubectl-mtv", "get", ...]) under the hood. No code changes to your CLI. No SDK integration. Just scan and serve.
From CLI to MCP

Establishing the SSE Channel

Step 1

curl -i -N http://localhost:8081/sse
HTTP/1.1 200 OK
Content-Type: text/event-stream
Connection: keep-alive

event: endpoint
data: /message?sessionId=session_9a7c3b21
Goal: Demystify how MCP over Server-Sent Events establishes a persistent connection. To connect to an HTTP-based MCP server, the client must first perform an HTTP GET request on the /sse route. The server keeps this connection open (Connection: keep-alive) and returns an event-stream. The very first message the server streams is an endpoint event containing the unique session path: /message?sessionId=session_9a7c3b21. From now on, the client will stream incoming events down this open GET channel, while sending outgoing JSON-RPC requests via standard HTTP POSTs to that custom /message endpoint.

Handshake and Tool Discovery

Step 2

# Initialize handshake
curl -X POST "$ENDPOINT" -d '{"jsonrpc": "2.0",
  "method": "initialize", "id": 1,
  "params": {"protocolVersion": "2024-11-05",
             "clientInfo": {"name": "demo", "version": "1.0"}}}'

# Discover available tools
curl -X POST "$ENDPOINT" -d '{"jsonrpc": "2.0",
  "method": "tools/list", "id": 2}'
Goal: Show the initialization handshake and tool discovery over POST. Once the SSE GET stream is active, we trigger our handshake by POSTing an initialize request to our session-based message endpoint. The server accepts the POST instantly, while the actual JSON-RPC replies are streamed asynchronously down the active GET channel we set up on the previous slide. Next, we send a standard tools/list request. The embedded Go MCP server replies over the active stream with the exact schema definitions for our tools, advertising its local execution capabilities to the model.

Goal: Directly contrast how the human operator and the machine assistant read the same resource outputs. Let's look at this side-by-side execution trace. On the left, we use the HTTP-based protocol. We POST a tools/call for mtv_read, passing get plans as our target command argument. On the right, we run the tool as a standard CLI plugin. We get clean, tabbed output formatted for a developer.
